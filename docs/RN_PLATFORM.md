# docs/RN_PLATFORM.md — React Native + Expo platform rules

The non-obvious rules of operating in this RN/Expo project. Read before
touching anything platform-shaped (native modules, build config, Metro,
Hermes, EAS).

## Expo managed workflow

- **No `ios/` or `android/` directory in git.** Those are generated by
  `expo prebuild`. Edits to generated native code are lost on the next
  prebuild, *silently*. This is one of the biggest sources of "it worked
  on my machine yesterday" for RN teams — and a top reason harness
  discipline matters here.
- **Native config goes through `app.json`** (and config plugins) — never
  directly into AndroidManifest.xml or Info.plist.
- **Native module decisions are versioned.** Before adding a native module,
  check the Expo SDK 54 list at
  https://docs.expo.dev/versions/v54.0.0/. If it isn't there, you need a
  config plugin or a development build. Note the decision in
  `claude-progress.md`'s Decisions log.

## Metro bundler

- **Cache lies sometimes.** If a change isn't visible, the answer is rarely
  "clear the cache." It is usually "you edited a file Metro doesn't watch."
  Check what Metro is configured to watch in `metro.config.js` before
  reaching for `--reset-cache`.
- **`require` cycles compile but break at runtime.** Hermes will throw a
  "ReferenceError: Cannot access X before initialization" at the bottom of
  the cycle, not at the top — which is misleading. The lint rule
  `import/no-cycle` is on for this reason.

## Hermes

- Hermes is the default engine on iOS and Android. **It does not have
  `Intl` everywhere** — date and number formatting work, but obscure locales
  may not. If a feature touches i18n, verify on both iOS and Android, not
  just web.
- Hermes stack traces are sourcemapped in dev but **not** in release unless
  you upload sourcemaps to EAS. If you see an obfuscated stack in a release
  log, that is the missing step, not a fundamental bug.

## File system

- **`fs` is not available.** Use `expo-file-system` for app data. The
  Node `path`/`fs` modules are bundled only because Metro shims a few, and
  they will throw at runtime if used.
- **Bundled assets are read-only.** `require('./feature_list.json')` works
  at bundle time and gives you a frozen object — that's how the home screen
  reads the feature list in `ui-001`.

## Lists

- Use `FlatList` or `@shopify/flash-list` for any list >20 items. Mapping
  over an array of `<View>`s mounts every cell synchronously and is the
  #1 cause of "scrolling is choppy" complaints. `ui-002` is verified
  against this rule.

## Reanimated + Worklets

- This repo includes `react-native-reanimated` v4 and `react-native-worklets`.
  Worklet functions must be marked `'worklet'` at the top. Calling JS
  functions from a worklet without `runOnJS` will crash silently on
  release builds.

## Platform splits

- Prefer `Platform.select` over `if (Platform.OS === ...)` chains. Easier to
  grep, easier to lint.
- For larger splits, use `Component.ios.tsx` / `Component.android.tsx` /
  `Component.web.tsx` — Metro picks the right one. Keep a `.tsx` fallback.

## EAS Build

- All builds go through EAS — `eas build --profile <preview|production>`.
- **Do not** commit `.env.local` or signing material; EAS Secrets owns
  those.
- Build numbers are managed by EAS (`autoIncrement: true` in `eas.json`).
  Don't bump them by hand.

## Performance budget (the bar for "feels native")

- Cold start to first interactive frame: < 2.5s on a mid-tier Android device.
- 60fps scrolling on the feature list (`ui-002`).
- Bundle size: < 5 MB JS for the main bundle. Above this, audit imports.

## What this means for agents

A change that touches the platform — adding a dep, changing `app.json`,
adding a config plugin, switching engines — is a **scope-changing** event.
It deserves its own feature in `feature_list.json`, not a side-edit.
